‚óŹ Go ‚ÄĒ and specifically because you already wrote the bot in Go.

  Why Go here:
  - Your existing discord-progress-tracker is Go, so the webhook sending, report
  formatting, and SQLite (tracker.db) logic ports directly. You reuse, not
  rewrite.
  - Single static binary per OS. You hand a coworker one .exe/binary, no runtime
  to install. Cross-compile all three from your machine
  (GOOS=windows/darwin/linux).
  - The two new pieces both have mature Go libs:
    - Screenshots: kbinani/screenshot (pure-ish, multi-monitor, Win/Mac/Linux).
    - System tray / background: fyne.io/systray (formerly getlantern) for a
  menu-bar icon.
  - Offline queue is trivial: write pending reports to SQLite with a sent flag,
  retry on a ticker when a webhook POST succeeds. You already have the DB.

  The one weak spot: Go's GUI story is thin. But you probably don't need a real
  window ‚ÄĒ a tray icon + native OS notification for the "what are you working
  on?" prompt is enough, and that's well-supported. If you do want real
  windows/dialogs, Fyne or Wails (Go + web frontend) cover it, at the cost of
  bigger binaries.

  Alternatives, quickly: Tauri (Rust+web) or Electron give nicer UIs but drop
  your Go reuse and bloat the binary. Python is fastest to prototype but
  packaging a screenshot-taking background app across 3 OSes (PyInstaller) is
  the exact kind of pain Go avoids. Not worth it given you're already in Go.

  Two things worth flagging before you build:

  1. macOS screenshots need Screen Recording permission (a system prompt the
  user must approve), and on Wayland Linux screenshots go through a portal ‚ÄĒ
  plan for "capture failed / not permitted" as a normal state, not a crash.
  2. Legal/consent: screenshot monitoring of employees has notice-and-consent
  requirements in many jurisdictions (EU/UK especially, some US states). Worth
  confirming coworkers are informed ‚ÄĒ both to stay legal and because covert
  monitoring, if discovered, usually costs more trust than the spotty-internet
  problem it's solving.

  Want me to sketch the structure (tray app + screenshot ticker + offline queue)
  reusing what's already in the repo?

‚ĚĮ How is the webhook configured?                                                

‚óŹ The webhook URL is a compile-time constant, not a runtime setting ‚ÄĒ
  deliberately, so the worker can't redirect it.

  How it's set: you pass WEBHOOK_URL to build.sh, which bakes it into the binary
  via -ldflags -X:

  WORKER_NAME="Alice" \
  WEBHOOK_URL="https://discord.com/api/webhooks/XXXXX/YYYYY" \
  ./build.sh

  That lands in internal/settings/settings.go:19 (var WebhookURL = ""),
  overwritten at link time.

  Where it's used: cmd/agent/main.go reads settings.WebhookURL in sendItem() and
  passes it to every discord.SendEmbed(...) / discord.SendImage(...) call.
  main() also refuses to start if it's empty (main.go:36).

  Where you get the URL: in Discord ‚ÄĒ Server Settings ‚Üí Integrations ‚Üí Webhooks
  ‚Üí New Webhook ‚Üí pick the channel ‚Üí Copy Webhook URL. Whatever channel you bind
  it to is where that worker's check-ins, updates, screenshots, and end-of-day
  report land. Point each worker's build at the channel you want their activity
  in (a shared channel, or one per worker).

  There's no .env or config file to edit ‚ÄĒ that's the point of the compile-time
  model. Changing the webhook means rebuilding that worker's binary.

  One caveat worth knowing: a Discord webhook URL is a bearer secret (anyone
  with it can post to that channel), and it's embedded in the binary ‚ÄĒ
  extractable by someone who reverse-engineers it. Same accepted ceiling as the
  AES key. If a URL leaks, rotate it in Discord (delete/regenerate the webhook)
  and rebuild.
